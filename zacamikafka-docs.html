AWS · Machine Image Pipeline

zacamikafka

A custom Amazon Linux 2023 AMI pre-loaded with Python, Java 21, and Apache Kafka prerequisites — with Systems Manager access baked in, so instances are reachable without SSH keys or open inbound ports. Built three ways: Terraform, AWS CLI, and Ansible.

Overview

Launching an instance from the zacamikafka image yields a Kafka-ready host with the JVM, Python client libraries, and the Kafka distribution already in place. Access is governed by an IAM instance profile rather than SSH credentials, which keeps the security group inbound-free.

Architecture

The pipeline launches a temporary builder from the latest base AMI, provisions it, snapshots it into the named image, then discards the builder. SSM access flows entirely over outbound HTTPS.

Figure 1 — Build and access flow. The builder is ephemeral; the AMI and IAM profile persist.

Build methods

Three supported paths. Expand each for the commands.

Terraformrecommended

Declarative, version-controlled build of the entire pipeline — base AMI lookup, IAM, security group, builder, and the resulting image.

shell 
terraform init      # download the AWS provider
terraform plan      # preview the resources
terraform apply     # build; AMI id is in the new_ami_id output
AWS CLI (manual)transparent

Run each step by hand for maximum visibility. Useful for learning and one-off builds.

shell 
BASE_AMI=$(aws ec2 describe-images --owners amazon \
  --filters "Name=name,Values=al2023-ami-2023.*-x86_64" \
  --query 'sort_by(Images,&CreationDate)[-1].ImageId' --output text)

aws ec2 run-instances --image-id "$BASE_AMI" \
  --instance-type t3.medium \
  --iam-instance-profile Name=zacamikafka-ssm-profile \
  --user-data file://userdata.sh

# after provisioning completes:
aws ec2 create-image --instance-id i-xxxx --name zacamikafka
Ansibleidempotent

Configuration-management approach; runs over SSH or the SSM connection plugin, then snapshot into the image.

shell 
ansible-galaxy collection install community.aws amazon.aws
ansible-playbook -i inventory_ssm.yml kafka-prereqs.yml

Terraform walkthrough

What the configuration does, block by block.

Resolving the base image

A data source queries Amazon for the newest Amazon Linux 2023 x86_64 HVM image, so every build starts from the latest patched base rather than a hard-coded, aging AMI ID.

IAM & SSM

An IAM role trusts the EC2 service and attaches the AWS-managed AmazonSSMManagedInstanceCore policy. The instance profile wraps that role; attaching it to an instance is what grants Session Manager access — no SSH ingress required.

Provisioning

The builder's user_data updates the OS, installs Python and Java, downloads Kafka into /opt/kafka, installs the Python clients, and enables the SSM agent. The aws_ami_from_instance resource then snapshots the result.

Production caveat aws_ami_from_instance can snapshot before user_data finishes — a real race condition. For reliable builds, prefer HashiCorp Packer (which waits for provisioning) or gate the snapshot behind an SSM RunCommand that polls for a completion marker.

Connecting via SSM

Any instance launched with the zacamikafka-ssm-profile attached is reachable without SSH.

  1. Attach the profileLaunch from the AMI with zacamikafka-ssm-profile as the instance profile.
  2. Confirm egressThe instance needs outbound 443 to the ssm, ssmmessages, and ec2messages endpoints (NAT, IGW, or VPC endpoints).
  3. Start a sessionRun the command below from your workstation.
shell 
aws ssm start-session --target i-xxxxxxxxxxxxxxxxx

Inside the session you can verify the build: java -version, /opt/kafka/bin/kafka-topics.sh --version, and python3 -c "import kafka".
